Currently, the processor will copy .ts files to the output directory. This is not what I want. I want .ts files to be processed by the TypeScript compiler and the resulting .js files to be copied to the output directory instead.

- you can assume that all .ts files are self-contained and do not have any external dependencies that need to be resolved

# Compile `.ts` Inputs to `.js` Outputs

## Summary
Update the file-processing pipeline so `.ts` inputs are no longer copied verbatim. Instead, each `.ts` file will be compiled with `tsc` and the emitted `.js` file will be written to the output directory, while all existing markdown/rss/copy behavior stays unchanged.

## Key Changes
- Add a new `TypeScriptFileProcessor : IFileProcessor` in `src/FileProcessors` and register it in DI from `src/Program.cs`.
- Add a small internal compiler service, for example `ITypeScriptCompiler`, responsible for invoking `tsc` via `System.Diagnostics.Process` and failing the run if compilation returns a non-zero exit code.
- Route `.ts` files in `src/FileProcessors/DispatchFileProcessor.cs` to `TypeScriptFileProcessor` before the default copy fallback.
- `TypeScriptFileProcessor` should:
  - derive the output path from the input stem, changing `.ts` to `.js`
  - invoke `tsc` for the single source file with explicit standalone-browser settings
  - emit directly into the requested output directory
  - never copy the original `.ts` file
- Use explicit compiler flags for the chosen mode:
  - `--module none`
  - `--target es2017`
  - `--outDir <outputDir>`
  - `--pretty false`
- Keep `CopyFileProcessor` unchanged for non-TypeScript assets.

## Public / Interface Changes
- No public CLI changes.
- New internal service registration:
  - `TypeScriptFileProcessor`
  - `ITypeScriptCompiler` implementation
- `IFileProcessor` stays unchanged; `.ts` handling is introduced only through dispatcher selection.

## Test Plan
- Add a small test project and wire it into the solution.
- Unit test dispatcher selection:
  - `.ts` resolves to `TypeScriptFileProcessor`
  - `.md`, post markdown, templates, `rss.json`, and ordinary assets still resolve exactly as before
- Unit test TypeScript processing:
  - input `foo.ts` produces output path `foo.js`
  - original `foo.ts` is not copied
  - compiler is invoked with the expected flags and output directory
  - compiler failures surface as exceptions and stop processing
- After implementation, run:
  - `dotnet format`
  - `dotnet build`
  - `dotnet test`

## Assumptions
- `.ts` files are standalone browser scripts, not module-based bundles.
- There is no `tsconfig.json`, so this feature will use fixed per-file compiler arguments rather than repo-driven TypeScript config.
- Input content does not intentionally include both `foo.ts` and a competing generated-source `foo.js` for the same output name.
- The application itself will not be run for verification, per repo instructions.
